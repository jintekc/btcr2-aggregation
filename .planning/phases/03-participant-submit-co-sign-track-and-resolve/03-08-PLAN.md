<objective>
Close the one remaining Phase 3 gap (03-VERIFICATION.md Truth 7, status gaps_found 6/7): anchor-status narration is not internally consistent across the components rendered on the participant's cohort page. Gap-closure plan 03-07 made the CompletionSummary Signed-line four-way honest (WR-01), but the SAME mode-honesty defect class persists in two adjacent render paths that 03-07 did not scope:

- Defect A (03-REVIEW.md WR-02): StageTimeline relabels its final row "Anchored" from `anchor?.enabled` alone (StageTimeline.tsx:130,152), so a broadcasting-but-unconfirmed or broadcast-failed live service shows an "Anchored" header while its own AnchorSubSteps and the CompletionSummary beside it honestly say broadcasting/failed. An internal contradiction on one page for the whole broadcasting window, or permanently on a failed broadcast.
- Defect B (03-REVIEW.md WR-01 reassertion): anchorSummaryState(null) returns 'hermetic' (participant.ts:689-691) and status is set to 'complete' synchronously before trackAnchor's first fetchAnchor read resolves, so a live service has a real, brief render window where the completion view falsely claims "This no-broadcast service does not publish to Bitcoin".

Both undermine Success Criterion 2 ("the participant sees...anchor status...in real time") and the phase's D-07 mode-honesty through-line.

Purpose: make the whole completion view narrate one honest anchor state at every moment, so the timeline header, the anchor sub-steps, and the completion summary never contradict one another (broadcasting, failed, anchored, or the pre-first-read checking window).
Output: a new neutral 'checking' member on the pure anchorSummaryState selector (with unit tests), a StageTimeline final-row label driven by anchorSummaryState === 'anchored' instead of the enabled bit, and a CompletionSummary that renders neutral confirming copy (Signed-line and round-trip placeholder) during the checking window instead of the hermetic no-broadcast copy.
</objective>
