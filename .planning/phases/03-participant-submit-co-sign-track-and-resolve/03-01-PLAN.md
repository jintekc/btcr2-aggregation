<objective>
Add the opt-in explicit-submit gate to the participant package: the deferred `onProvideUpdate` mechanism (Finding 1) that lets the browser hold the awaited update callback until the user clicks "Submit my DID update" (D-12), while every headless caller keeps today's auto-submit. This is the PART-03 mechanism the web store (03-04) drives; nothing here renders UI.

Purpose: Turn the auto-submit into a user-consented submit without a library change and without breaking a single existing caller.
Output: An additive-optional `onSubmitGate` on `CreateParticipantOptions`, a `SubmitGateInfo` type carrying the pre-built body, and specs proving both paths.
</objective>

## Phase Goal (user story, derived from the ROADMAP Phase 3 goal)

**As a** participant who joined a cohort by choice, **I want to** submit my DID update, take part in the cohort's n-of-n MuSig2 co-signing round, track the anchor, and resolve my updated DID, **so that** I complete the real aggregation lifecycle from the cohort I discovered, not a standalone demo stepper.

> Note: the ROADMAP `**Goal:**` line is outcome-shaped rather than literal user-story form; this story is derived from that goal and the four success criteria. MVP mode is all-or-nothing for the phase; this plan is the thin backend mechanism the Wave 3 UI slice (03-05) needs to make explicit submit real.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| participant runner <- onProvideUpdate callback | The library awaits an app-controlled callback that decides whether/when to submit the signed update |
| in-browser key -> signed update | BIP340 signing happens with the participant's key; the built body must be the submitted body |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01-01 | Denial of Service (protocol) | onSubmitGate awaited callback | medium | mitigate | A never-resolved gate stalls only until the service's existing `phaseTimeoutMs` expires the cohort (Finding 2); no new server surface. The store (03-04) clears the deferred without settling on teardown so a stopped runner never leaves a dangling reject/resolve. |
| T-03-01-02 | Tampering | signed update body | high | mitigate | Build the update exactly once and submit/compare that captured object; never rebuild (BIP340 non-determinism). getSubmittedUpdate is the single source for the round-trip check. |
| T-03-01-03 | Repudiation | cooperative non-inclusion | low | mitigate | The decline (return null) path runs BEFORE the gate, so a baked-mismatch identity declines explicitly rather than being silently dropped; getDeclineReason records why. |

No new packages are installed in this plan (zero-install phase); supply-chain checkpoint not applicable.
</threat_model>
