## Phase Goal (user story)

**As a** participant co-signing a cohort where one member drops mid-round, **I want to** have the cohort still complete via a k-of-n script-path fallback instead of failing outright, **so that** a single defector cannot deny the whole cohort its anchor - while n-of-n MuSig2 stays the normal, cheaper, more private primary spend. (Closes the optional k-of-n fallback item under UAT finding F1.)

<objective>
Close the F1c item of UAT finding F1: keep n-of-n MuSig2 as the PRIMARY spend and ADD the ADR-042 k-of-n script-path fallback for liveness on a signing stall. The library (`@did-btcr2/aggregation@0.4.0`) fully supports this fallback: the service option `autoFallbackOnStall` wires the phase-stall timer to `triggerFallback`, the participant runner auto-approves a `fallback-requested` (via the default `onApproveSigning`) and signs the k-of-n script path, and `signing-complete` reports `path: 'script-path'`. The app already commits the fallback tapleaf into every beacon address (`beacon-address.ts`), and the dashboard already renders the `fallback` status + the `script-path` badge (`dashboard.ts`, `CohortCard.tsx`, `web/src/lib/types.ts`). What is missing is purely activation: the SERVICE never sets `autoFallbackOnStall`, never calls `triggerFallback`, and `buildCohortConfig` never sets `fallbackThreshold` (repo-wide source grep = 0), so a stalled cohort FAILS instead of falling back.

This plan flips that on. `createService` gains an `autoFallbackOnStall` option threaded to the runner; the demo server opts in by default (env `AUTO_FALLBACK=0` to disable), so the self-hostable product recovers liveness on a signing stall. `buildCohortConfig` gains an optional `fallbackThreshold` (default undefined -> the library's n-1). Genuine k-of-n PRIMARY signing is NOT offered by the package and is not attempted. The default hermetic path stays deterministic: with no stall the outcome is still the optimistic n-of-n key path.

Cross-check with plan 02-06 (F2): `autoFallbackOnStall` only converts a SIGNING-phase stall into a fallback; an idle Advertised cohort stall still expires (handled by 02-06's surfaced expiry). The two are distinct and documented against 02-06's discovery-window timers.

Output: `autoFallbackOnStall` on `CreateServiceOptions` (threaded to the runner) in `index.ts`, the `AUTO_FALLBACK` opt-in in `demo-server.ts`, the optional `fallbackThreshold` on `buildCohortConfig` (+ spec) in `packages/shared`, and a new hermetic `e2e/fallback-cohort.ts` (+ `e2e:fallback` script) proving both the key-path default and the forced-stall script-path fallback.
</objective>
