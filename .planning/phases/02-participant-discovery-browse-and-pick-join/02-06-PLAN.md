## Phase Goal (user story)

**As an** operator who advertises a cohort and waits for strangers to discover it, **I want to** have my advertised cohort stay live long enough to be found and joined over time, and to see it (not have it silently vanish) if it does expire, so I can re-advertise it, **so that** the two-sided discover-over-time loop actually works instead of every cohort disappearing about a minute after I advertise it. (Closes UAT finding F2 under the PART-01/PART-02 discoverability gap.)

<objective>
Close UAT finding F2. Today an advertised, unjoined cohort is torn down after ~60s by the library's per-phase stall timer (`phaseTimeoutMs` default 60000): an idle cohort never leaves the Advertised phase, so the stall timer fires, `session.removeCohort()` fully deletes it, and `operator-cohorts.ts` prunes its enrichment map - the cohort silently vanishes from the participant directory, `/v1/status`, and the operator list, and is unjoinable thereafter. The 60s default was tuned for the removed booth topology (in-process fillers joined within seconds) and defeats the two-sided discover-over-time North Star.

The fix is two-fold. First, give an advertised cohort a lifetime appropriate for strangers to discover and join over time: raise the `phaseTimeoutMs` and `cohortTtlMs` defaults to a generous discovery window (30 minutes), still tunable via the existing `PHASE_TIMEOUT_MS` / `COHORT_TTL_MS` env. The library exposes only ONE inter-phase stall timer (there is no option to exempt the Advertised phase), so raising it is the clean library-native lever; the documented tradeoff is that a genuine mid-signing stall also waits longer before the runner acts - which plan 02-07 (F1c) converts from a hard failure into a graceful k-of-n script-path fallback. Second, surface expiry instead of silently deleting: retain a terminal `expired` record (the cohort's config + the rejection reason) when a cohort's completion rejects, list it to the operator as `state: 'expired'`, and let the operator re-advertise it with a gated `POST /v1/operator/cohorts/:id/readvertise` (operator-driven, consistent with D-17).

Purpose: an operator-advertised cohort stays discoverable long enough for a stranger to browse and join it by choice, and an operator is never left staring at a cohort that silently disappeared.

Output: the raised timer defaults in `demo-server.ts` (+ the CreateServiceOptions JSDoc in `index.ts`), the terminal `expired` record + `readvertiseExpired` in `operator-cohorts.ts`, the gated re-advertise route in `hono-adapter.ts`, the operator client + store + list surfacing in `packages/web`, and the extended `e2e:operator` proving expiry is surfaced and re-advertise works.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator -> gated POST /v1/operator/cohorts/:id/readvertise | Re-advertise is a mutating operator action; it must inherit the session gate + same-origin CSRF check like advertise, never be reachable anonymously. |
| service -> participant directory | An expired cohort is operator-only state; it must never appear as a joinable participant directory entry. |
| terminal-record map | An app-side map keyed by cohort id; unbounded growth would be a memory-exhaustion vector. |
